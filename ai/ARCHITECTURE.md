# Architecture (MVP)

`PROJECT.md` describes SWARM's long-term vision: a stateless GCP cloud orchestrator (Cloud Run + Pub/Sub + Firestore) talking gRPC to a local daemon. **For the MVP, skip the GCP layer entirely** and mirror Cascade's architecture instead — a router + queue + worker running locally (Docker Compose, same as Cascade), reachable from GitHub via a Cloudflare Tunnel instead of a cloud ingress service. The gRPC/cloud-engine design in `PROJECT.md` is not being built yet; treat it as a possible future phase once the local-only version works end-to-end for one user, not as the current target. If this changes, update this file and `README.md` in the same change.

## Why copy Cascade's shape instead of `PROJECT.md`'s

`PROJECT.md` was written before we had a concrete reference implementation to copy. Cascade already solves "webhook comes in → figure out which project/task it belongs to → run an agent → report back" for exactly this kind of pipeline, in the same language, and its author knew Node.js well. Rebuilding that shape from scratch (or building the gRPC/Pub/Sub version) is strictly more work for no MVP-stage benefit. SWARM's two product differences from Cascade are:

1. **PM provider = GitHub Projects**, not Trello/JIRA/Linear (Cascade explicitly has no GitHub Projects adapter — this is net-new).
2. **SCM = GitHub**, same as Cascade — copy this piece close to verbatim.

Everything else — router/queue/worker split, manifest-based provider registration, credential scoping, loop prevention — should be copied, not reinvented.

## Components (MVP)

```
GitHub (repo + Projects v2)
   │  webhooks (pull_request, pull_request_review, issue_comment, issues, check_suite, projects_v2_item)
   ▼
Cloudflare Tunnel (external, not our concern — just a public HTTPS URL pointed at the router)
   ▼
Router  (Hono HTTP server, local Docker container)
   — verifies webhook signatures
   — resolves which SWARM project the event belongs to
   — records a durable dispatch (Postgres, ADR-002) and publishes its BullMQ
     wake-up (PR review-lifecycle jobs are prioritized ahead of PM-board jobs,
     so a review never queues behind a planning or implementation run —
     `src/queue/producer.ts`'s `priorityFor`)
   ▼
Worker  (host process — NOT containerized, one job at a time or a small pool)
   — looks up the trigger handler for the event
   — provisions a Git worktree (see "Worktree lifecycle" below)
   — spawns `claude` or `antigravity` CLI with the worktree as CWD
   — validates structured agent hand-offs and performs deterministic SCM delivery
   — persists step progress, then posts results back to GitHub Projects

Dashboard (Hono HTTP server + tRPC, host process)
   — exposes a `/health` check and mounts a tRPC API under `/trpc`
   — self-hosted, single-process model running locally on the host
   — binds to `127.0.0.1`; authorizes every request except `/health` and `ping` by a per-user session cookie (issue #281 task 2), then scopes every project-keyed procedure to the caller's project membership (issue #281 task 4)
   — serves the built `web/dist` SPA statically as a fallback when present (self-hosted mode)
```

Redis (for BullMQ) and Postgres (for project config, credentials at rest, and run history — same role it plays in Cascade) run in the same Docker Compose stack as the router. The **worker and the dashboard are the exceptions**: they run directly on the host rather than in containers — the worker because it provisions Git worktrees and spawns the `claude` / `antigravity` CLIs (which need the developer's own PATH, auth, and config), and the dashboard (scaffolded via Hono and tRPC) for local-first, single-process execution. They connect to Redis/Postgres over their published host ports (`REDIS_URL` / `DATABASE_URL` in `.env`). There is no separate "cloud" process for the MVP; router, worker, dashboard, Redis, and Postgres are all local.

### Observability

All code logs through one shared logger (`src/lib/logger.ts`) — a deliberately tiny, dependency-free wrapper (no pino/winston) mirroring Cascade's `logger.info/warn/error/debug(message, context)` shape. It emits **structured JSON** (one `{level,time,msg,...context}` object per line) for machine parsing, or — for a human watching a terminal — a local-clock timestamp, a colored level tag, the bound `component` as a `[component]` prefix, and remaining context as `key=value` pairs (color only on a real TTY, never when `NO_COLOR` is set). Format follows `SWARM_LOG_FORMAT` (`json`|`pretty`) and auto-picks pretty on an interactive TTY, json when piped/containerized — **leave it unset** in `.env` rather than hardcoding either value: the router's own `docker-compose.yml` environment block already pins it to `json` for that container regardless of `.env`, so setting it in `.env` only ends up overriding the host worker's auto-detection (and silently losing pretty/color) without changing the router at all. `SWARM_LOG_LEVEL` (`debug|info|warn|error`, default `info`) sets the minimum emitted level. Because router and worker share a log stream, each entry point calls `configureLogger({ component })` at startup so every line it emits is tagged `component: "router"` / `"worker"` and stays attributable.

## Provider abstraction

Two integrations, each following the manifest/registry pattern from `ai/CODING_STANDARDS.md`:

### SCM: GitHub (`src/integrations/scm/github/`)

The integration itself lives under `src/integrations/scm/github/` — consistent with the PM provider (`src/integrations/pm/github-projects/`) and the module shape in `ai/CODING_STANDARDS.md` — but its internals are ported close to verbatim from Cascade's `src/github/*`, so match Cascade's file shapes (`scm-integration.ts`, `client.ts`, `personas.ts`) rather than reinventing them. The router adapter is router infrastructure, so it lives under `src/router/adapters/github.ts` (mirroring Cascade), not inside the integration folder.

- Dual-persona tokens (`implementer`, `reviewer`) scoped via `AsyncLocalStorage` (`src/integrations/scm/github/client.ts`'s `withGitHubToken`), never passed as plain arguments. Token references live in the project config's `credentials` block; the secrets themselves are resolved from Postgres per-persona (`src/config/provider.ts` → `src/db/repositories/credentialsRepository.ts`).
- Router adapter (`src/router/adapters/github.ts`) parses `pull_request`, `pull_request_review`, `issue_comment`, `issues`, and `check_suite` events, resolves the SWARM project from the repo, and dispatches with the right persona's credentials in scope.
- Loop prevention via an `isSwarmBot(login)` check (`src/integrations/scm/github/personas.ts`) on comment events, exactly as described in `ai/CODING_STANDARDS.md` — a persona never reacts to its own ack/reply comments; PR/review lifecycle routing between personas is handled by `getPersonaForLogin`, not by the drop gate.
- Pipeline code requests deterministic mutations through the provider-neutral `ScmDeliveryProvider` seam (`src/scm/delivery.ts`). Agents write Zod-validated semantic hand-offs and never commit, push, create PRs, submit reviews, or post comments. The GitHub adapter supplies persona-scoped implementations, durable delivery markers, existing-PR recovery, and safe expected-SHA pushes. Step progress is written beside the hand-off in the retained worktree so a retry resumes the failed delivery operation rather than repeating completed mutations.
- Merging an approved PR goes through the provider-neutral `ScmMergeProvider` seam (`src/scm/merge.ts`, issue #253; direct-merge-only per issue #292): a single `mergePullRequest(project, prNumber, approvedHeadSha)` capability returning a terminal `MergePullRequestOutcome` — `merged`, `not-ready`, `not-eligible`, `policy-blocked`, `unsupported`, or `provider-error` — never a thrown error, so a refusal can't retroactively fail an already-completed Review. Pipeline code never merges: after an eligible Review approval the worker persists a **durable merge dispatch** (`src/worker/merge-automation.ts`, see "Durable dispatch state machine" below) whose executor invokes this capability. GitHub is the first implementation: every attempt re-reads the PR's current state, verifies the approved head SHA and approval still hold, and performs the **direct** REST merge (`pulls.merge` via `src/integrations/scm/github/client.ts`'s `mergePullRequestDirect`) under the project's implementer PAT, pinned to the approved head so a concurrent push produces GitHub's non-merged stale-head response rather than merging unreviewed code. GitHub's native auto-merge is deliberately never requested (issue #292): it is unavailable on many private repositories, has no portable equivalent in other SCMs, and its error text (`Auto merge is not allowed for this repository`) once bypassed merge automation entirely. GitHub-specific classification (which REST error means what — a merge-queue requirement is `unsupported`, 405/409 are `not-ready`, 403 is `policy-blocked`) stays inside the adapter. A future Bitbucket/GitLab adapter implements the same interface against its native direct PR/MR merge API and outcome mapping, with no dispatch or worker changes.

### PM: GitHub Projects (`src/integrations/pm/github-projects/`) — net-new, no Cascade equivalent

GitHub Projects (v2) is GraphQL-only; there is no REST equivalent for reading/writing item fields. Shape the provider around:

- **Work item** = a Projects v2 item (usually backed by an Issue or PR in the repo).
- **Status** = the item's single-select "Status" custom field — moving a task through the pipeline (Backlog → Planning → In Progress → In Review → Done) means updating that field's option value via `updateProjectV2ItemFieldValue`.
- **Events** = the `projects_v2_item` webhook event (`created`, `edited`, `reordered`, `deleted`), filtered to status-field edits — this is the equivalent of Cascade's `pm:status-changed` trigger. It is **never** a repo-level event: GitHub delivers it on an **organization** webhook or to a **GitHub App** with the Projects permission (there is no plain user-account webhook). For SWARM's user-owned board the App route is effectively the only option — see `docs/github-projects-v2-api.md` §5.
- **Comments** = GitHub Projects items don't have their own comment thread; write agent output (e.g. Antigravity's plan) as a comment on the linked Issue/PR, not as a Projects-native comment.
- **Assignees** = the backing Issue/PR's assignees, read in the same round-trip as the item and mapped to the provider-neutral `WorkItemAssignee` (`handle`/`displayName`/`providerId`) on every `WorkItem` — GitHub's `login` vocabulary stops at the adapter. `assignees` is always present (`[]` when nobody is assigned), and the capability flag `PMProvider.supportsAssignees` lets a provider with no assignee concept opt out explicitly, exactly like `supportsDependencies`. SWARM resolves a handle to one of its own users through the identity-linking seam (below), never by pattern-matching a GitHub identity.

Implement it against the same `PMProvider`-shaped interface Cascade uses (`getWorkItem`, `moveWorkItem`, `addComment`, `listWorkItems`, …) so the router/trigger/dispatch code stays provider-agnostic — see Cascade's `src/pm/types.ts` for the exact interface to mirror.

## Pipeline phases

Per `PROJECT.md` §5, adapted to GitHub Projects as the board and GitHub as the sole SCM:

Each worker-spawned agent is scoped to one phase. A shared prompt guard forbids the manual
`solve-issue` skill, arbitrary subagents/skills, and later-phase work; only the worker owns
phase hand-offs, personas, and worktrees.

**Automation opt-in gate (issue #131).** The two board-driven phases below — the phases that *start*
SWARM working on a work item — only dispatch for an item carrying the project's
`pipeline.automationLabel` (default `swarm`). The check is provider-neutral: `resolveAutomationLabel`
/ `hasAutomationLabel` (`src/pm/automation-label.ts`) match the configured name against
`WorkItem.labels`, the canonical PM capability every adapter already populates from its own native
labels — no `PMProvider` method was widened, and a Trello/Jira/Linear adapter inherits the gate for
free. It is evaluated at the single dispatch choke point (`processJob`, `src/worker/consumer.ts`),
*before* the in-flight guard and `acquireProjectSlot`, so a skip costs no slot, no worktree, and no
tokens; because it lives there it is re-evaluated on every fresh webhook, delayed retry,
self-enqueued next phase, capacity promotion, reconciler republish, and manual "Retry now" —
removing the label therefore stops all later dispatches, but never terminates an agent already
running. A skip logs the missing label and settles the dispatch `completed` with outcome
`skipped-not-eligible` (a retried run row is finalized `failed` with the same reason rather than left
`deferred`). That outcome is deliberately named for eligibility in general, not for the label:
#339's worker-authorization gate lands at the same call site and settles through the same value with
its own reason. Split children Planning creates carry the configured label so SWARM's own siblings
aren't gated out of its pipeline. An explicitly empty `automationLabel` disables the gate. The label
is an automation opt-in, **never** an access-control mechanism — ADR-001's authorization layers are
separate and unaffected. The SCM continuation phases (3–5) are not gated yet (phase 2/2 of the same
work item).

1. **Planning** — item enters "Planning" status → worktree provisioned → the planning agent writes `proposed_plan.md` → plan posted as a comment on the linked Issue. By default the item stays in "Planning" (a human reviews the plan, then moves it to "ToDo" to greenlight Implementation); `pipeline.planning.autoAdvance` flips that to an automatic move. When `pipeline.planning.autoSplit` is on (default) and the agent judges the item too large for one PR, it also writes `proposed_split.json`: the original item is re-scoped into the smaller **first** task (`proposed_plan.md` is that task's plan) and the remaining work is spawned as sibling items via `PMProvider.createWorkItem`. Each sibling ends in **Planning**, is tagged `swarm:split-child`, and gets a comment explaining the split. SWARM first creates its card in Backlog, writes the parent-generated validated preplanned marker (`<!-- swarm-preplan:v1 … -->`, `src/pipeline/preplan.ts`) into its issue body, then moves it to Planning. That ordering guarantees both the Planning-move event and any delayed card-creation event find a valid marker; the trigger suppresses the redundant Planning run while the marker remains valid and the item keeps its `swarm:split-child` label. A malformed/stale marker, `swarm:replan`, or removing the label falls back to normal Planning. This keeps splitting behind the provider-agnostic `PMProvider` (no GitHub Projects specifics leak into the phase). A split also **chains the phases as dependencies** (issue #330): the agent orders the titles by phase (`Phase 1/N …`), and for each spawned sibling the phase records a cumulative `blocked by` relationship on *every* preceding phase (phase 3 ← phases 1 and 2; phase 4 ← 1, 2, 3; …) — via `PMProvider.addBlockedBy` (behind `supportsDependencies`, so a provider that can't model dependencies degrades gracefully) *and* by naming those blockers in the sibling's split comment. The two guards mean a later phase is never startable before its predecessors land.
2. **Implementation** — item enters "ToDo" → **a dependency gate runs first** (issue #330): before the "In Progress" move, the worktree, or the agent, the phase asks the `PMProvider` for the item's still-open blockers (`listBlockers` — native `blocked by` relationships plus prerequisites named in the item's description/comments) and, if any are unfinished, **defers** the run as a token-free `recheck` (no worktree, no agent → zero model tokens; re-checked every `SWARM_DEPENDENCY_RECHECK_MS` on its own budget, separate from the rate-limit budget, until the blocker closes or `SWARM_DEPENDENCY_MAX_WAIT_MS` is exhausted, at which point it settles `failed` with an "issue X must be done first" comment). This is what stops a later phase from being implemented before its predecessors land (the out-of-order build that produced the PR #326/#327 conflict). Otherwise → the phase moves it to "In Progress" to report the pickup → the agent implements/verifies and writes a semantic hand-off → SWARM validates the prepared tree, commits/pushes, opens or reuses the PR, and links it back to the item. Successful delivery moves the item to "In review" exactly when `pipeline.review.enabled` is on; otherwise it remains "In progress".
3. **Review** — PR opened / checks pass (independent of the Projects status report) → the reviewer agent writes a structured verdict and body → SWARM submits the formal review under the reviewer persona. A durable, restart-safe ledger (`review_verdicts`, one row per `(project, repository, PR, head SHA)`, issue #235) caps every PR at **two** submitted verdicts — an initial review plus at most one **re-review**: the `pr-review` trigger reserves a slot before dispatching (reusing a same-head retry's existing slot rather than charging it again) and refuses a third; the phase marks its reservation `submitted` once GitHub confirms the review (idempotently, so a crash between delivery and this write is repaired by a retry without a duplicate submission). **The re-review is scoped, not a fresh full pass (issue #328):** when the phase finds a prior `request-changes` verdict already submitted for this PR (`getPriorSubmittedReview`), it builds the re-review variant of the prompt (`buildReviewPrompt(..., isReReview=true)`), which tells the reviewer to verify *only* whether the previously requested changes were implemented correctly — approve when they were, otherwise request changes with a strong, specific fix instruction — and **not** to surface newly-noticed pre-existing issues the first review missed (the only admissible new findings are regressions the fix commits themselves introduced). This keeps the last permitted verdict spent on confirming the fix rather than restarting the cycle over unrelated work. Only a submitted `approve` requests a merge (`pipeline.respondToReview.autoMerge`, issue #235) — the worker's Review success path persists one durable merge dispatch per approving run (`requestMergeAutomationIfEligible`, `src/worker/consumer.ts` → `src/worker/merge-automation.ts`, issue #292), and the dispatch's executor performs the direct PAT merge through the provider-neutral `ScmMergeProvider`, retrying transient `not-ready` outcomes on a bounded backoff (15s doubling to 5min, six retries). Every outcome lands on the Review run's `review_merge_*` columns; a terminal refusal (`not-eligible`, `policy-blocked`, `unsupported`, `provider-error`, `retry-exhausted`) is visible there and on the dispatch record but never fails the completed Review.
4. **Respond to review** — by default, the reviewer persona's `changes_requested` review → the implementer addresses the batch and writes a structured response → SWARM commits/pushes any fix and posts the response idempotently. `pipeline.respondToReview.skipOnMinors` defaults to true so approvals and comment-only reviews do not consume a separate agent run; set it false to respond to every reviewer-persona verdict. The trigger resolves the submitted review's ledger record (by review id, falling back to PR/head for the narrow webhook race before the Review phase has stored it) and, on the safety cap's **second** `changes_requested` verdict, stops here instead of dispatching — that PR needs a human, and the outcome is recorded as `manual-intervention-required` rather than starting an unbounded third cycle; a lookup failure for a `changes_requested` event fails the same way (closed) rather than risk reopening the cycle on a persistence outage. Respond-to-review's own outcomes (`fixed`/`pushed-back`/`no-findings`) never request a merge themselves — addressing feedback isn't a fresh approval. Like Implementation, it reports progress on the board as a status *report*, not a trigger: the card moves to "In progress" while the implementer works and back to "In review" once it has responded (best-effort — a card it can't resolve or move is logged and skipped, never failing the response). Neither status starts a phase, so bouncing the card can't re-fire Review; only a new fix commit does.

**Reliable follow-up Review after a `fixed` push (issue #241).** A `fixed` outcome whose commit was actually pushed and differs from the reviewed head reliably enqueues exactly one follow-up Review for the new SHA — reusing the same Review trigger, aggregate-check routing (`decideCheckSuiteOutcome`), reviewer persona, PR/head dispatch dedup, and durable two-verdict ledger a real `check_suite` webhook would drive, via a synthetic `check_suite` `completed` event (`src/pipeline/follow-up-review.ts`) carrying the PR number/branch/repo and new head. Unlike `selfEnqueueNextPhase`'s best-effort PM hand-off, this is part of the phase's own deterministic delivery: it runs after the push/comment delivery steps and before the delivery-progress checkpoint (`DeliveryProgress.followUpEnqueued`) is saved, so a queueing failure raises `DeliveryDeferredError` (preserving the worktree for a resumed retry) instead of silently dropping the follow-up, and the enqueued job's id is a deterministic hash of `(project, PR, new head)` so a crash between a successful enqueue and that checkpoint write can't duplicate the job. `pushed-back`, `no-findings`, a failed/unpushed response, and an unchanged head (the pushed commit somehow matching what was reviewed) enqueue nothing. A follow-up Review that approves may still request the same approval-only merge described above; Respond-to-review itself still never requests it.
5. **Respond to CI** — a PR's check suite completes with a failing check → worktree on the PR's task branch → Claude Code (implementer persona) inspects the failing checks, fixes the build surgically and pushes (or reports no code change was warranted) — mirrors Cascade's `respond-to-ci` agent. A per-PR fix-attempt cap stops a never-sticking fix from looping (each fix commit is a new SHA, so it re-triggers CI).

### Trigger wiring (SWARM-53)

The phases are self-contained orchestrators; the trigger registry (`src/triggers/`) is what connects inbound events to them. `registerBuiltInTriggers` (`src/triggers/builtins.ts`) registers three handlers under `src/triggers/handlers/`:

- **`pm-status-changed`** — one handler for both PM-driven phases. On a `projects_v2_item` Status edit (or a card added), it re-reads the item authoritatively via the `PMProvider` (never trusting the webhook's Status value — `docs/github-projects-v2-api.md` §5) and, per `resolvePipelinePhaseForOptionId`, starts Planning (→ Planning) or Implementation (→ ToDo), or returns `null` for any other status — including "In progress", which Implementation itself moves an item *to* as a status report, not a trigger. The Board view's drag-and-drop actually fires a `reordered` action (no `changes.field_value` at all — a real delivery confirmed this), not `edited`, so the adapter accepts both; a Redis-backed dedup (`pm-status-dedup.ts`) stops the resulting harmless within-column reorders from re-dispatching the same phase.
- **`preplan-invalidated`** — starts fallback Planning for a split child whose card is already in Planning when an `issues` webhook reports a body edit, addition of `swarm:replan`, or removal of `swarm:split-child`. Those issue changes do not emit a Projects Status event. The handler lists Planning items through `PMProvider`, matches the event's backing-item URL, and re-evaluates the authoritative marker/labels before dispatching; a still-valid preplan remains a no-op.
- **`pr-review`** — the PR-lifecycle handler. Starts Review on a non-draft, **same-repo**, **SWARM-authored** PR opening or its checks passing, and routes a *failing* check suite to Respond-to-CI. A `review` disposition also reserves the durable two-verdict safety-cap slot (see Pipeline phases §3) after the dispatch dedup claim; `respond-to-ci` never consumes one. On a `check_suite` completion it re-queries *every* check on the head SHA (`getCheckSuiteStatus`) rather than trusting the single suite's own conclusion — GitHub fires one event per workflow — and decides via `check-suite-decision.ts`: review if all complete and none failed, **respond-to-ci** if a check failed, or **defer** if a check is still running, scheduling a coalesced ~30s recheck (`scheduleCoalescedJob`) that re-queries fresh state (a `recheckAttempt` cap bounds the loop against a stale Actions API). Fork PRs are dropped from the Review path (their head SHA is unreachable for the detached review checkout). An **author-persona gate** reviews only PRs authored by a SWARM persona (the implementer opens every SWARM PR), mirroring Cascade's `decideCheckSuiteGates` default `authorMode='own'`, so a human- or third-party-bot-authored PR never burns a review; on the `check_suite` path the gate runs *before* the aggregate query (one `pulls.get`, since that payload carries no author) so a PR we'd never review doesn't also pay the heavier Actions-API call. The respond-to-ci dispatch reuses the same per-SHA dedup slot (review and respond-to-ci are mutually exclusive for a commit) and adds a per-PR fix-attempt cap (`respond-to-ci-attempts.ts`) so a fix that never makes CI green can't loop forever.
- **`resolve-conflicts`** — a side-car handler for merged pull requests. It lists open same-repository, SWARM-authored PRs targeting the advanced base, retries GitHub's asynchronous mergeability result with bounded coalesced jobs, and dispatches only confirmed conflicts. Redis dedup keys each attempt by PR/head/base state; the implementer phase merges the base normally and checks the remote head before pushing.
- **`pr-review-submitted`** — starts Respond-to-review on a `pull_request_review` `submitted` with `changes_requested`, authored by the *reviewer* persona (`getPersonaForLogin`). `pipeline.respondToReview.skipOnMinors: false` restores dispatch for approve/comment verdicts. Stops instead of dispatching on the safety cap's second `changes_requested` verdict (see Pipeline phases §4).

A matched handler returns a `TriggerResult` naming the phase plus its resolved inputs; the worker's `processJob` (`src/worker/consumer.ts`) dispatches on that phase and calls the matching `runXPhase`, building the concrete GitHub Projects `PMProvider` (`src/integrations/pm/github-projects/provider.ts`) for the two board-driven phases. The review handler is guarded by a cross-process dedup claim (`src/triggers/review-dispatch-dedup.ts`, SWARM-62): a Redis `SET NX EX` keyed on `owner/repo:pr:headSha`, claimed before dispatch and left to expire by TTL (nothing releases it on failure in the MVP), so a PR that opens *and* then passes checks (or emits several check-suite successes) is reviewed once per commit rather than once per event. It fails closed — a claim it can't obtain (duplicate, or Redis down) skips the dispatch — and the respond-to-ci path shares that same slot before its own attempt cap runs.

`processJob` also self-enqueues the hand-off from Planning's `autoAdvance` to Implementation, rather than waiting on GitHub to echo that move back as a webhook (`selfEnqueueNextPhase`, `src/worker/consumer.ts`). The router's loop-prevention gate (`GitHubProjectsRouterAdapter.isSelfAuthored`) correctly drops *every* Projects status-change webhook authored by a SWARM persona — but Planning moves its own card to "ToDo" using the `implementer` persona, the very identity that gate checks against, so the direct webhook for that move is dropped every time by design. Left as webhook-only, the pipeline only ever advanced when some other, differently-attributed event for the same item happened to arrive later — unbounded, and observed live leaving an item in "ToDo" for hours with Implementation never dispatching. Instead, once a PM-driven phase's own move lands on a status that starts the next phase (`resolvePipelinePhaseForStatusKey`, `src/pm/pipeline.ts`), the worker enqueues a synthetic `github-projects` job for the same item directly onto the BullMQ queue (`enqueueJob`, no `deliveryId` — the same "synthetic/manual" carve-out `src/queue/producer.ts` already documents), which re-enters the normal trigger-match → authoritative-re-read → dedup path a real webhook would. Best-effort: a failure to self-enqueue is logged, not thrown, so it can't turn an already-succeeded phase into a failed job.

## Worktree lifecycle

Unchanged from `PROJECT.md` §4 — this part of the original spec is SWARM-specific and already correct:

- Main repo at `~/swarm/{project-name}/`, worktrees under `~/swarm/{project-name}/.swarm-workspaces/task-<id>/`.
- `git worktree add` per task; config/caches and the `cascade` sibling-checkout pointer (`.env`, root and nested package dependencies such as `node_modules` / `web/node_modules`, `cascade`, build caches) grafted in via symlinks with **absolute** targets — a relative link would dangle at the worktree's `.swarm-workspaces/<name>/` depth (see `ai/RULES.md` §1).
- `git worktree remove --force` on completion.
- Stale `task-<id>` worktrees left behind by interrupted runs are reclaimed by a retention sweep running periodically in the background of the worker process, or manually on-demand via `swarm worktrees prune`. The sweep preserves up to a configurable number of most-recently-active worktrees (defined by `worktreeRetention.maxWorktrees` in the project config, defaulting to 10), checking against a Redis-backed "worktree lease" and local cleanliness (`git status`) to ensure in-flight and dirty worktrees are never pruned.

## Harness (agent-CLI execution engine)

Once the worker has a provisioned worktree, it hands off to the harness (`src/harness/agent-cli.ts`, `runAgentCli`) to actually run the agent. The harness is deliberately narrow: it spawns `claude` or `antigravity` (Node's `child_process.spawn`, no subprocess library) with the worktree as CWD, streams stdout/stderr line-by-line (optional callbacks + `logger.debug`) while accumulating the full output, and resolves with `{ exitCode, signal, stdout, stderr, durationMs, timedOut }`. A non-zero exit is a normal outcome the caller inspects — only a spawn failure (e.g. the CLI isn't installed) throws. `timeoutMs` and an `AbortSignal` both kill the run (SIGTERM, escalating to SIGKILL). Prompt construction, persona/token selection, and the queue→worktree→harness→cleanup lifecycle live in the worker (SWARM-17), not here.

**Capability-aware target routing (issue #346).** A phase's agent config holds an *ordered* list of `targets` (`agents.<phase>.targets`, `src/config/schema.ts`), and the worker runs the highest-priority one whose CLI this machine can actually run: `selectTarget` (`src/worker/target-selection.ts`) walks the list against the CLIs discovery found runnable (the `cli_quotas` snapshot, `src/harness/quota-discovery.ts`), and `agentOverrideFor` (`src/worker/consumer.ts`) feeds the chosen target's `cli`/`model`/`reasoning` into the unchanged four-tier model chain. A target with no `cli` (the phase's coded default) is always eligible; when no target's CLI is available — or capability discovery never ran, or the lookup fails — the preferred target is used anyway, so a genuinely missing CLI still fails visibly on spawn rather than silently skipping the phase. A per-run pinned retry (`cliOverride`/`modelOverride`/`reasoningOverride`) bypasses routing. Availability is loaded once per job, best-effort, alongside the global defaults. This is the **single local worker** choosing among its own installed CLIs — ADR-001's federated scheduler (worker enrollment, sharing consent, assignee affinity, deferring until a capable worker is free) is a separate, unbuilt layer.

**Model + reasoning mapping (issue #180).** The harness takes a *logical* `model` plus a normalized `reasoning` level and turns them into per-CLI launch args through `resolveModelLaunch` (`src/harness/models.ts`), the one place provider-specific reasoning semantics live: Claude gets `--model <alias> --effort <level>`, Codex gets `--model <id> -c model_reasoning_effort="<level>"`, and Antigravity — which has no reasoning flag — gets the reasoning folded back into the combined `agy models` variant string it passes to `--model` (e.g. logical `gemini-3.5-flash` + `high` → `"Gemini 3.5 Flash (High)"`). A pre-#180 config that stored the combined string directly is recognized and passed through verbatim, so it keeps launching that exact variant. Omitting reasoning preserves each CLI's own default (no flag for Claude/Codex; the model's default variant for Antigravity). `src/harness/models.ts` is the versioned capability catalog — logical models, their supported reasoning levels, and known defaults — cached in code, never discovered by invoking a model.

### Durable dispatch state machine (issue #284, ADR-002)

Every attempt to start or resume a pipeline phase is one row in the Postgres
`dispatches` table — the single source of truth for pending/in-flight
orchestration state (`docs/decisions/ADR-002-durable-dispatch-state-machine.md`).
BullMQ jobs are pure wake-ups carrying a `dispatchId`; the worker acts only
after atomically claiming the dispatch (`pending`/`retry-scheduled` → `leased`),
so a cancelled, completed, or superseded dispatch refuses every late delivery —
redelivery, delayed retry, slot release, or reconciliation. The lifecycle is
`pending → leased → running → completed/failed`, with `retry-scheduled` for
deferred attempts (the derived retry payload is persisted on the row *before*
any queue work, so a crash never loses retry intent) and `cancelled` for
user/operator cancellation. Wake-up job ids are deterministic per
(dispatch, wake sequence), so the reconciler's re-publish is a queue no-op when
the wake-up already exists. Project-capacity waits are `pending` dispatches with
wait reason `project-capacity`, woken by slot releases under the
continuation-priority policy — there is no separate Redis registry. A worker
startup + periodic reconciler (`src/dispatch/reconciler.ts`) reclaims expired
leases (failing the dispatch and its still-`running` run together), re-publishes
lost wake-ups, and — once, at startup — imports legacy shapes (Redis
pending-continuation entries, deferred runs with no active dispatch, and
pre-#292 `not-ready` merge-follow-up intent) as durable dispatches.
Merge automation is the one agent-less dispatch kind (issue #292): a
`merge-automation` payload (dedup key `merge:<reviewRunId>`, linked to the
approving Review run) skips triggers, worktrees, and project slots entirely —
`processJob` routes its claimed wake-up straight to
`processMergeAutomationDispatch`, which merges via the `ScmMergeProvider` and
settles the dispatch itself (`completed` with a `merge-*` outcome, `failed` on
a provider error, or `retry-scheduled` while transiently `not-ready`).

The dashboard deliberately exposes two complementary read models (issue #313):

- **Queue is dispatch-centric.** The Queue API/UI (`runs.queued`) reads every waiting
  dispatch from Postgres — state, wait reason, priority, and scheduled time — never a
  BullMQ snapshot. `swarm queue clear` cancels those canonical records first. A fresh
  dispatch with no `runId` is Queue-only because no persisted attempt exists yet.
- **Runs is run-centric.** The Runs list (`runs.list` / `listRunsFromDb`) reads persisted
  attempts by their normal lifecycle, but hides a `deferred` attempt linked to a pending
  or retry-scheduled dispatch to avoid displaying a duplicate row (issues #279/#316).
  Deferred attempts with no waiting dispatch remain visible in Runs as history and for operator recovery.

### Failure handling & rate-limit retries (issue #91)

A phase whose agent exits non-zero throws, and `processJob` turns that into a `phase-failed` `JobOutcome` rather than rethrowing — an agent run isn't idempotent, so a BullMQ retry storm is worse than surfacing the failure (the phase already logged the agent's stdout/stderr, and the `attempts: 3` default only ever fires for the infra throws that happen *before* the agent runs). The exceptions are transient failures: a **usage/session-limit** hit where the agent never did any work, an **aborted** run where the worker itself shuts down mid-phase, or a deterministic-delivery failure after the agent has already produced a valid checkpoint. Delivery deferrals preserve the worktree and expose the wrapped push/hook/API cause in the run rather than misreporting it as a worker shutdown.

`src/harness/agent-failure.ts` classifies a failed run (`classifyAgentFailure` → `rate-limit` | `capacity` | `timeout` | `aborted` | `stalled` | `error`). On a `rate-limit`, `capacity`, `aborted`, or genuinely-interrupted `timeout` error, `processJob` returns a `phase-deferred` outcome with a computed delay instead of `phase-failed`; the worker entrypoint (`src/worker/index.ts`) re-enqueues the job (delayed, fresh job id, `rateLimitRetryAttempt` bumped) via `enqueueDelayedRetry`, capped by retry budgets. The retry delay is floored above the review-dispatch-dedup TTL.

`capacity` is a transient **provider-capacity** signal, matched per CLI so one CLI's provider error isn't triggered by another merely quoting it: Codex's structured `error` / `turn.failed` events containing `selected model is at capacity` anywhere in captured output (or its textual terminal banner), and Claude/Claude Code's terminal `529 Overloaded` / `overloaded_error` banner (Anthropic's documented temporary-overload response — issue #229). It defers on a shorter capacity budget (`MAX_CAPACITY_RETRIES`, `MIN_RETRY_DELAY_MS` backoff) and retries **fresh, not resumed** — the provider rejected the request before the model did any work, so there is no partial reasoning or edit to continue. Only a bare status number never matches; human-readable provider banners use a conservative terminal-output window, so borrowed CI logs or reviewed code mentioning these numbers stay a plain `error`.

A retry **reuses the originating `runs` row** rather than inserting a second one (issues #136 and #146): the `phase-deferred` outcome carries the run's `runId`, the dispatch settle path (`settleDispatchRetry`, `src/worker/consumer.ts`) persists it in the derived retry payload on the dispatch record, and on the next claim `processJob` resets that same row to `running` (clearing `completedAt`/`error`/`nextRetryAt` and the outcome columns) instead of creating a new one. A fresh webhook that intentionally reruns a deferred or failed task/phase also claims and resets its latest terminal row, so automatic retries, manual retries, and user-triggered reruns all remain one logical dashboard run. A **manual "Retry now"** (the `runs.retryNow` tRPC mutation, offered for both `deferred` and `failed` runs) never flips the run row itself — the run turns `running` only when the worker actually claims the dispatch, so a failed enqueue can't strand a false `running` run (issue #284). It re-opens the run's active dispatch for an immediate attempt with `rateLimitRetryAttempt` reset to 0 (bypassing the automatic cap), folding any cli/model/reasoning overrides into the stored payload; a run with no active dispatch (terminally `failed`, or a legacy orphan) gets a fresh dispatch reconstructed from the row's persisted `jobPayload`. The conditional dispatch transition plus the one-active-dispatch-per-run unique index prevent concurrent automatic/manual retries from starting the same row twice.

PM retry dispatch intent is separate from Implementation branch reuse. Manual and automatic retries
carry `resumePmPhase` so an item already reporting "In progress" still re-enters its original phase,
while `implementationBranchProvisioned` is persisted only after successful task-worktree acquisition.
The retry uses the existing branch only when that explicit checkpoint is present; a run row by itself
never proves that provisioning happened.

### User-initiated termination (issue #166)

A **running** or **deferred** run can be terminated from the dashboard (the `runs.terminate` tRPC mutation). The dashboard and worker are separate processes, so termination never touches a PID: `terminate` records the intent in a durable, run-id-keyed Redis set and publishes a notification (`src/queue/cancellation.ts`). For a **running** run the worker aborts the agent via the per-run `AbortController` threaded into `runPhase` (the same SIGTERM→SIGKILL path shutdown uses), so the phase runs its normal worktree/lease cleanup and settles the row as `failed` with a user-termination reason — `handlePhaseFailure` checks the cancellation set to tell a user termination apart from a worker-shutdown abort (which still defers). For a **deferred** run `terminate` cancels the canonical dispatch record first (`cancelDispatchForRun` — a cancelled dispatch refuses every future claim, so no retry, slot release, or reconciliation can resurrect the work; issue #284), then atomically flips the row `deferred → failed` (`markRunUserTerminated`); if a concurrent automatic pickup wins that race, the run is now `running` and the cancellation flag drives the worker to terminate it instead. Keying on the immutable run id (not the task id) means a later retry of the same task is never terminated by an old request, and `retryNow` clears the flag before re-running. The mutation is idempotent: an already-terminal run returns its settled state rather than erroring. Captured logs are preserved (the aborted agent's output is stored like any failed run); only the board "failed" comment is skipped, since an intentional stop isn't a stall to investigate.

On a **terminal** `phase-failed` (such as `error`, `timeout`, or `stalled`), `processJob` also posts a failure comment on the backing Issue for the work-item-carrying phases (planning/implementation) via `reportPhaseFailureToBoard` → `pm.addComment`, so a human watching the board sees *why* the item stalled. If the failure kind is `stalled` or `timeout` (the agent gave up waiting for a model response), the posted comment appends a hedged splitting suggestion, recommending that the task be split by hand into smaller pieces before re-triggering. The comment is best-effort and skipped for PR-driven phases (review/respond-*), which have no board work item.

## Single-user scope

No per-org config and no multi-tenant Firestore layer. One Postgres row per SWARM project, one set of GitHub credentials (per persona) per project. The dashboard is loopback-only (`127.0.0.1`) and, as of #281 task 2, gated by per-user session auth (below) rather than a shared secret; as of #281 task 4 each request is then scoped to the caller's project membership (below), so an authenticated user no longer sees every project. As of #281 task 5 a project may *opt in* to a `discoverable` visibility that layers a limited public-discovery view and a request/approve join flow on top of that strict members-only default (below), without wiring any execution/routing permission to visibility. Revisit the per-project credential model as remaining ADR-001 work (worker registration/routing) lands.

**Multi-user foundation (ADR-001, issue #281) — in progress.** The identity domain landed first: a `SwarmUser` Zod schema (`src/identity/schema.ts`), a `users` table (`src/db/schema/users.ts`), a provider-neutral identity read-model service (`src/identity/service.ts`), the installation-level `instanceAdmin` flag, and a `swarm users` operator CLI. A SWARM user is a person who signs in to this installation — **not** an SCM identity and **not** an implementer/reviewer GitHub credential.

**Session auth (task 2) is now live.** The shared `DASHBOARD_TOKEN` bearer guard is gone: a user gets a password via `swarm users set-password` (stored as a `scrypt` salt+hash in `users.password_hash`, never plaintext), signs in at `/login`, and receives an opaque session token — stored only as a SHA-256 in `user_sessions`, delivered as an HTTP-only, `SameSite=Strict` cookie (`Secure` off localhost), never in a response body or log. The tRPC context (`src/api/trpc.ts`) resolves the caller from that cookie (`src/identity/auth.ts` `resolveSession`); every procedure except `ping` is an `authedProcedure` that 401s without a session, and logout revokes it. Because every request now carries a cookie via `credentials: 'include'`, the dashboard fronts all routes with a credentialed CORS layer (`src/lib/cors.ts`, `CORS_ORIGIN`, default `http://localhost:5173`) so the documented separate-origin dev setup (SPA on Vite, API on `DASHBOARD_PORT`) can send it — inert for the recommended same-origin deploy, which never pre-flights.

**Project membership (task 3, layer 2) is the read model — data only, not yet enforced.** It adds a `ProjectMembership` Zod schema and per-project role (`projectAdmin | member | contributor`, ordered by privilege) with `canAdministerProject`/`canWriteProject`/`canReadProject` predicates (`src/identity/membership.ts`), a `project_members` table (`src/db/schema/projectMembers.ts`, unique per `(project, user)`, cascade-deleting with either its project or its user), a repository (`src/db/repositories/projectMembersRepository.ts`), a provider-neutral membership read-model service (`src/identity/membership-service.ts`, whose `listAccessibleProjectIds` resolves an installation admin to *all* project ids and any other user to their membership set), and a `swarm members` operator CLI to seed membership before the dashboard UI exists. Membership is created directly (repository/CLI/admin) or, as of task 5 (below), via an approved open-project join request. It is the read model that **project-scoped authorization (task 4, now live — below)** and **open-project policy (task 5, now live — below)** build on.

**Project-scoped authorization (task 4, layer 2) is now enforced.** `src/api/authz.ts` turns the membership read model into the API's enforcement layer — the second of ADR-001's three authorization layers (installation role → project membership → worker enrollment). Every project-keyed tRPC procedure (`routers/projects.ts`, `routers/runs.ts`, nested `routers/credentials.ts`) calls `assertProjectAccess(user, projectId, minRole)` or scopes a cross-project list through `filterAccessibleProjects` / `accessibleProjectScope`. An `instanceAdmin` bypasses every check (layer 1 override); a user with no membership cannot even learn a project exists (reads and mutations alike surface `NOT_FOUND`, never `FORBIDDEN`, so real project ids never leak); a member below the required role gets `FORBIDDEN`. The role → capability matrix, defined once in `authz.ts` and mapped from the `canReadProject`/`canWriteProject`/`canAdministerProject` predicates:

| Capability | Min role | Procedures |
| --- | --- | --- |
| Read a project, its runs, its (masked) credentials | `contributor` | `projects.getById`/`list`, `runs.list`/`queued`/`getById`/`getLogs`/`getOutput`, `credentials.list` |
| Drive a project's runs | `member` | `runs.retryNow`/`terminate`/`putBack` |
| Administer config, credentials, or delete the project | `projectAdmin` | `projects.update`/`delete`, `credentials.set`/`delete` |

Creating a project (`projects.create`) is open to any authenticated user, and the creator is written a `projectAdmin` membership row in the same call, so they can administer what they just made without an operator seeding membership first. Host-level procedures that are not keyed to a project (`ping`, `auth.me`, `settings.*`, `scm.verifyGithubToken`, `quota.*`) stay `authedProcedure` only — installation-admin gating of global settings is a separate, later concern, not project-scoped authorization.

**Open-project policy + join/request flow (task 5) — the final foundation slice.** Task 4's default is strictly members-only; task 5 adds an *opt-in* public face without loosening it. It keeps four permissions deliberately separate — **discovery ≠ joining ≠ worker registration ≠ task routing** — so public visibility never equates to execution permission (ADR-001 access model). A per-project `visibility` policy (`src/config/schema.ts` `ProjectVisibilitySchema`: `private` default | `discoverable`, a `text` column on `projects`) drives it:

- **Discover** — `projects.listDiscoverable` (any authenticated user) returns a **limited** read of `discoverable` projects the caller cannot already access: `id` + `name` only, projected in the query itself (`listDiscoverableProjectsFromDb`) so credentials/config/repo/run internals never even leave the DB. It excludes the caller's own projects; an `instanceAdmin` (who already sees everything) gets nothing new.
- **Join / request** — `projects.requestMembership(projectId)` files a **pending** membership request (`project_membership_requests` table, mirroring `project_members` with a `pending|approved|rejected` status and a *partial* unique index allowing one pending request per `(project, user)`). This resolves ADR-001 open question #1 in favour of **request/approve over immediate membership**: joining an open project grants no access on its own. A private/unknown project is `NOT_FOUND` (existence never leaks); an existing member or pending request is `CONFLICT`.
- **Approve / reject** — `projects.listMembershipRequests`/`approveMembershipRequest`/`rejectMembershipRequest` reuse `assertProjectAccess(user, projectId, 'projectAdmin')`, so only a `projectAdmin`/`instanceAdmin` acts on requests (a non-member gets `NOT_FOUND`, a lower-role member `FORBIDDEN`). Approval creates a `contributor` (**read-only**) `project_members` row atomically with resolving the request, and is idempotent w.r.t. an existing higher role.

Critically, **nothing in this task wires visibility or membership to worker registration (#132) or task routing (#130)** — those remain distinct, still-unbuilt permissions (ADR-001's third layer). The highest role joining ever yields is `contributor`, which grants read and nothing more.

**Registered worker identity (#132 Phase 1) — data + CLI only, inert.** Where a `SwarmUser` is *who* a person is and a `ProjectMembership` is *what* they may do on a project, a **worker** is *where* a user can execute — a locally operated execution environment they own (ADR-001's third identity layer). It adds a `Worker` Zod schema (owner user id, a safe machine `displayName`, and declared CLI `capabilities` reusing `AgentCliSchema` — `claude | antigravity | codex`; `src/identity/worker.ts`), a `workers` table (`src/db/schema/workers.ts`, FK to `users` `ON DELETE CASCADE`, unique per `(owner, displayName)`, unique credential hash), a repository (`src/db/repositories/workersRepository.ts`), a provider-neutral `worker-service` (`src/identity/worker-service.ts` — credential issue/hash primitives, `registerWorker` returning the raw credential **once**, `refreshWorkerCapabilities`, and the `resolveWorkerByCredential` authentication seam), and a `swarm workers` operator CLI (register/list/set-cli/remove) to register workers before any dashboard UI exists. The **worker credential** is the worker's own authentication material — distinct from an SCM PAT — stored only as a SHA-256 (`workers.credential_hash`), never in the `Worker` read model and returned raw exactly once at registration, mirroring `users.password_hash` / `user_sessions.token_hash` and `src/identity/auth.ts`. Like the rest of the foundation it is inert: nothing reads it yet — worker **sessions/leases** (Phase 2) and project **enrollment/routing** (Phase 3, #130) do.

**Worker session leases (#336 Phase 2) — one live claim per worker, fenced.** On top of Phase 1's identity, a **worker session** is the single live claim on a registered worker's execution environment, so two `swarm-cli` daemons can never drive the same machine at once. It adds a `WorkerSession` Zod schema (`src/identity/worker-session.ts` — a per-worker monotonic **fencing token**, `lastHeartbeatAt`, a nullable `currentRunId`), a `worker_sessions` table (`src/db/schema/workerSessions.ts`, FK to `workers` `ON DELETE CASCADE` with a **unique index on `worker_id`**, `released` flag, and `current_run_id` → `runs.id` `ON DELETE SET NULL`), a repository whose `acquireLease` is an atomic conditional transaction (`src/db/repositories/workerSessionsRepository.ts` — lock the row `FOR UPDATE`; reject a *live* lease, replace an *expired* or *released* one with a bumped token, else insert the first; the row lock plus the unique index guarantee exactly one concurrent caller wins), and a provider-neutral `worker-session-service` (`src/identity/worker-session-service.ts` — `acquireSession`/`heartbeat`/`releaseSession`/`setCurrentRun`, each authenticating the worker via its Phase-1 credential through `resolveWorkerByCredential`, plus the `validateFencingToken` seam and `getLiveSessionForWorker` read). A session **expires** after `SWARM_WORKER_HEARTBEAT_TTL_MS` (default 60s) of no heartbeat or is gracefully released (flagged with `released: true`): re-acquisition increments the per-worker fencing token across releases, and a heartbeat/run-attach carrying the old (replaced/released) token is rejected — the standard fencing-token guard a later dispatch/advance (#130) validates against before it lets a worker act. The same user may hold independent live sessions for *different* registered workers; leases are keyed by worker, not by user. Still in-process (the MVP worker has no daemon/network transport) and not yet wired to routing/enrollment.

**Worker-project enrollment, sharing consent, and rosters (#337 Phase 3) — the routability seam + read models.** On top of Phase 1's identity and Phase 2's sessions, an **enrollment** links a worker to a project with project-scoped execution constraints and the owner's revocable consent — ADR-001's third authorization layer (installation role → project membership → worker enrollment). It adds a `WorkerEnrollment` Zod schema (`src/identity/worker-enrollment.ts` — `status` (`pending|active|suspended`), `allowedClis` (a subset of the worker's capabilities), `concurrencyAllocation`, and an owner-controlled `sharingConsent`) plus the predicate `isRoutable(e) = e.status === 'active' && e.sharingConsent` — the named seam the #130 dispatch gate consumes (the enrollment analogue of `canReadProject` in `membership.ts`). A `worker_project_enrollments` table (`src/db/schema/workerProjectEnrollments.ts`, FKs to `workers` and `projects` both `ON DELETE CASCADE`, unique per `(worker_id, project_id)`, `sharing_consent` default false), a repository (`src/db/repositories/workerEnrollmentsRepository.ts`), and a provider-neutral `worker-enrollment-service` (`src/identity/worker-enrollment-service.ts`) back it. The service owns the writes (`enrollWorker` — validating `allowedClis` is a subset of the worker's capabilities; `approveEnrollment`; `setSharingConsent`; `setEnrollmentStatus`; `updateEnrollmentConstraints`) and two **secret-free** read models: `listProjectRoster(projectId)` (every enrolled worker — display name, owner, capabilities, status, allowed CLIs, concurrency, sharing consent, `isRoutable`, and derived busy/current-run) and `listOwnerWorkers(ownerUserId)` (the owner self-service view). **Busy/current-run is derived from run lifecycle, never client-supplied**: `deriveWorkerRunState` reads the worker's live Phase-2 session `current_run_id` and reports busy only when that run is actually `running` in `runs`. The read models assemble their views by naming safe fields explicitly, so no repo path, PAT, local CLI token, or credential hash can ride along. The tRPC `workers` router (`src/api/routers/workers.ts`) exposes owner self-service procedures scoped to `ctx.user` (`listMine`/`enroll`/`setConsent`/`updateConstraints`) and project-roster procedures gated by `assertProjectAccess` exactly like `routers/projects.ts` — `contributor` reads the roster, only `projectAdmin` approves (`approveEnrollment`) or revokes/reactivates (`setStatus`), with NOT_FOUND-vs-FORBIDDEN existence hiding — and the `swarm workers` CLI gains `enroll`/`approve`/`consent` for operator seeding. **Revoking sharing consent or suspending the enrollment flips `isRoutable` to block *future* automatic dispatch; it never terminates a running agent** — the actual dispatch gate and process teardown are #130, out of scope here. Project isolation falls out of the roster query being keyed on `projectId`: a worker enrolled in project A never appears in B's roster.

**Assignee resolution + identity linking (#130 Phase 1) — the affinity lookup, inert.** ADR-001's routing rules make assignment an *execution-affinity* rule: a work item assigned to a PM identity that maps to a SWARM user may run only on that user's eligible worker. That needs two things neither the PM nor the identity domain had, and this slice adds exactly those, changing no dispatch behaviour. **(1) A provider-neutral read of who an item is assigned to**: `WorkItemAssignee` + `WorkItem.assignees` + the `PMProvider.supportsAssignees` capability flag (`src/pm/types.ts`), populated by the GitHub Projects adapter from the backing Issue/PR (see "PM: GitHub Projects" above). Because `trigger.workItem` is already built from an authoritative `pm.getWorkItem()` (`src/triggers/handlers/pm-status.ts`), assignees flow to the trigger layer for free. **(2) A SWARM identity-linking seam**: a `UserIdentity` Zod schema (`src/identity/user-identity.ts` — a `(provider, handle)` pair owned by one `users` row), a `user_identities` table (`src/db/schema/userIdentities.ts`, FK to `users` `ON DELETE CASCADE`, **unique per `(provider, handle)`**), a repository (`src/db/repositories/userIdentitiesRepository.ts`), and the resolver `src/identity/assignee-resolver.ts` (`resolveUserForAssignee(assignee, provider)` / `resolveAssignedUser(workItem, provider)`), whose `provider` argument is the caller's `PMProvider.type` so the seam stays provider-agnostic. This answers ADR-001 open question 5 by construction: a handle maps to **at most one** SWARM user (the unique index), keys are normalized (trimmed + lowercased, so casing never splits a link), and an unlinked handle resolves to `undefined` rather than a guess. Links are seeded by an operator (`swarm identities link`) — automatic linking (OAuth / SCM account discovery) and a dashboard UI are deliberately out of scope. Like the rest of the foundation it is inert: no phase, trigger, or scheduler reads it yet — the eligibility gate that consumes it is the later #130 phase.

**Worker eligibility evaluation (#338, #130 Phase 2) — the pure half of the dispatch gate.** `evaluateWorkerEligibility` (`src/identity/worker-eligibility.ts`) folds ADR-001's routing prerequisites into one structured verdict: `{ eligible: true }` or `{ eligible: false, reason }`, where `IneligibilityReason` is `missing-enrollment` (no enrollment, or one not yet `active`) | `missing-consent` (consent never granted or revoked) | `worker-unavailable` (no live session, or the enrolled `concurrencyAllocation` fully used) | `missing-cli-capability`. Checks run in the ADR's order — active enrollment → sharing consent → connection/health → free capacity → declared CLI capability — and the **first** missing signal is returned, so a caller has one reason to show rather than a set; the first two checks are `isRoutable` split apart only so a revoked consent reads as `missing-consent`. It is **target-scoped, never target-choosing**: it judges exactly one candidate target from `agents.<phase>.targets` against that target's effective CLI (its own `cli`, else the phase's coded `DEFAULT_*_CLI` passed as `phaseDefaultCli`) and requires that CLI to be both a declared worker capability *and* allowed by the enrollment — it never reorders or falls back across targets, so Phase 3 can call it once per target and keep configured target priority across workers instead of collapsing back to a single model. The module is dependency-light and side-effect-free (like `src/pm/dependencies.ts`): the DB-backed signals arrive as a caller-resolved `WorkerAvailability` (`connected` from `getLiveSessionForWorker`, `activeRuns` derived from run lifecycle). Nothing dispatches through it yet — the dispatch wiring, scheduler ordering, assignee affinity, and the scheduler-only `assignee-worker-unavailable` reason are Phase 3.
