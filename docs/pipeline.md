# Pipeline and architecture overview

This guide explains the user-visible pipeline and its security boundaries.
For implementation-level details, see [`ai/ARCHITECTURE.md`](../ai/ARCHITECTURE.md)
and the durable-dispatch ADRs under [`decisions/`](decisions/).

## Pipeline phases

Each phase's agent CLI, model, and reasoning level are configurable per project, per phase, via `swarm.config.json`'s `agents` block (`src/harness/models.ts` is the catalog of valid `cli`/`model` combinations and the reasoning levels each model supports) — the descriptions below are the code's own defaults when a project doesn't override them. Planning's `autoAdvance` setting defaults to **off**, so a human reviews the plan and moves it to "ToDo". Implementation's completion status is not independently configurable: it moves the item to "In review" exactly when `pipeline.review.enabled` is on, and otherwise leaves it "In progress".

**Automation is opt-in per work item.** Before SWARM starts either board-driven phase (Planning, Implementation) it checks that the item carries the project's `pipeline.automationLabel` (default `swarm`); an item without it is skipped with a log naming the missing label and the dispatch outcome `skipped-not-eligible` — no worktree, no agent, zero model tokens. The check runs at every dispatch, so removing the label stops the item's later phases too, though it never terminates a run already in flight. It is matched against the work item's labels through the provider-agnostic PM interface, so it works the same on any board. The label is an automation opt-in, **not** an access-control mechanism — it grants nobody access to anything. See [`configuration.md`](./configuration.md#project-config-swarmconfigjson).

Worker-spawned agents are scoped to exactly one phase. Their prompts prohibit the manual `solve-issue` skill, arbitrary subagents/skills, and continuing into another phase; the worker owns every phase hand-off and supplies the correct persona and worktree. The manual skill remains available for explicitly requested interactive sessions outside the worker.

1. **Planning** (Claude Code by default) — item moves to "Planning" → the worker provisions a detached worktree, the planning agent reads the code graph and writes `proposed_plan.md` → plan is posted back to the board. Moves the item to "ToDo" itself if `autoAdvance` is on for this project; otherwise a human moves it once they've reviewed the plan. If `autoSplit` is on (default) and the agent judges the item too large for a single PR, it **splits** it: the original item becomes the smaller first task (re-scoped, possibly renamed), and the rest ends as sibling items in **Planning**, tagged `swarm:split-child`, and carrying a comment explaining the split. SWARM creates each child in Backlog only long enough to write the parent's validated child-specific plan; it moves the card to Planning after that marker exists, so both the move event and a delayed creation event suppress the redundant Planning dispatch. A malformed/stale marker, `swarm:replan`, or a removed split-child label runs Planning normally; repo-level `issues` events dispatch that fallback even while the card remains in Planning. If marker/status preparation fails, the child remains in Backlog and its comment reports that fallback honestly. A human starts those siblings' implementation in order. Each split also carries an ordered **`Phase X/N`** title and is chained as a dependency: SWARM records a cumulative `blocked by` relationship on every earlier phase (and lists them in the split comment), so a later phase can't be implemented before its predecessors land. The plan is held to a minimal-scope rule and a deterministic scope gate: the agent records why the work is one task (in `proposed_scope.json`), and an unsplit plan that declares two or more independent concerns fails Planning with a request to narrow or split rather than advancing an oversized plan to Implementation.
2. **Implementation** (Claude Code by default) — item moves to "ToDo" → **a dependency gate runs first**: if the item is still `blocked by` an unfinished prerequisite (a native relationship or one named in its description/comments), the run is **deferred** and re-checked cheaply — no worktree, no agent, **zero model tokens** — until the blocker closes (then it proceeds automatically), or it gives up after `SWARM_DEPENDENCY_MAX_WAIT_MS` and fails with an "issue X must be done first" comment. Otherwise → the phase reports the pickup by moving the item to "In progress" → the agent implements and verifies the change, then writes a structured hand-off. SWARM validates the prepared tree, commits and pushes it under the implementer persona, reuses or creates the PR, and links it back to the board. Successful delivery moves the item to "In review" when Review is enabled; with Review disabled it remains "In progress".
3. **Review** (Claude Code, reviewer persona) — a PR-opened or successful check-suite lifecycle event dispatches Review independently of the Projects status transition → the agent writes a structured verdict and review body; SWARM submits the formal review under the reviewer persona. When `pipeline.respondToReview.autoMerge` is enabled, a submitted `approve` persists a durable **merge dispatch** (issue #292): an agent-less unit of dispatch work that merges the PR **directly** under the project's implementer credential, pinned to the reviewed head SHA. GitHub's native auto-merge is never requested — it is unavailable on many private repositories and has no portable equivalent in other SCMs. The dispatch's outcome (merged, waiting, or a terminal refusal) is persisted on the Review run and shown on its dashboard detail page. A transient `not-ready` result — including GitHub still converging on the review it just submitted — is retried on the dispatch's bounded backoff (a handful of attempts over a few minutes), surviving worker/Redis restarts and cancellable like any queued dispatch; each retry re-checks the PR's current state and approval, so a changed head or an overridden approval reports `not-eligible` instead of merging stale content. Retry exhaustion, and every other terminal outcome (`not-eligible`/`policy-blocked`/`unsupported`/`provider-error`), stays visible on the run but never fails the completed review.
4. **Respond to Review** (Claude Code, implementer persona) — the agent processes the review and leaves fixes plus a structured reply; SWARM commits/pushes any fix and posts one idempotent response comment.
5. **Respond to CI** (Claude Code, implementer persona) — the agent diagnoses CI and leaves a surgical fix or no-fix decision; SWARM performs any commit/push and posts the explanation. A per-PR attempt cap stops a fix that never turns CI green from looping forever.
6. **Resolve Conflicts** (Claude Code, implementer persona) — the agent resolves conflicting intent and leaves the verified tree; SWARM rechecks the observed remote head, creates the normal merge commit, pushes without force, and posts the result.

Deterministic delivery uses durable operation identities and a step-progress sidecar in the retained worktree. Equivalent PRs, reviews, and comments are detected and reused, so a failed delivery step can resume without duplicating already-completed external mutations.

## Security model

- **Per-user session auth, localhost-bound** (issue #281 task 2): The dashboard binds to `127.0.0.1` rather than `0.0.0.0`, and every request except `/health` and the `ping` probe is authorized by a per-user session — a password login mints an opaque token delivered as an HTTP-only, `SameSite=Strict` cookie (`Secure` off localhost); only its hash is stored, and logout revokes it. This replaced the earlier shared-secret `DASHBOARD_TOKEN` bearer guard.
- **Project-scoped authorization** (issue #281 task 4): on top of session auth, every project-keyed tRPC procedure is scoped to the caller's project membership (`src/api/authz.ts`). An ordinary user sees and acts only on projects they are a member of — read needs `contributor`, driving runs needs `member`, and administering config/credentials or deleting a project needs `projectAdmin`; a non-member gets `NOT_FOUND` (existence hidden), a member below the required role gets `FORBIDDEN`. An `instanceAdmin` overrides all of it and sees every project. See `ai/ARCHITECTURE.md` for the full role → capability matrix.
- **Open-project policy: discovery ≠ joining ≠ execution** (issue #281 task 5): a project may opt in to `visibility: discoverable`, which exposes a **limited** public read (`projects.listDiscoverable` — id + name only, never credentials/config/repo/run internals) to any authenticated user and lets them file a **pending** membership request (`projects.requestMembership`). A `projectAdmin`/`instanceAdmin` approves it into a `contributor` (read-only) membership or rejects it — joining an open project grants no access on its own (resolving ADR-001 open question #1 for request/approve over immediate membership). Discovery and joining are deliberately kept separate from worker registration (#132) and task routing (#130): public visibility never equates to execution permission, and the highest role a join ever yields is `contributor`.
- **Dual-persona GitHub credentials** (MVP): separate implementer/reviewer tokens, borrowed from Cascade's loop-prevention model — a persona never reacts to its own output. Stored in Postgres, not a cloud secret manager (see `PROJECT.md` §6.1).
- **Zero-knowledge codebase, still**: everything above is local — router, worker, Postgres, Redis. Nothing leaves the machine except webhook payloads and PR/comment metadata.
- *(Future, once `PROJECT.md` §2.2's cloud engine exists)*: board/SCM API keys move into Google Cloud Secret Manager; the daemon authenticates to the cloud via a bearer token + device hardware ID (`PROJECT.md` §6.2).
